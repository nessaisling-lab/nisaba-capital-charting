## Phase 2: "Fetch This Ticker" Button (v5.0.2)

**Goal:** User clicks a button in the dashboard, scraper binary runs for that one ticker, data appears.

**Architecture:** Subprocess spawning via `std::process::Command`. Dashboard finds the scraper binary adjacent to its own executable. Scraper gets a new `--ticker AAPL` CLI mode that runs a subset of the pipeline for one ticker and exits.

### 2a. Scraper: CLI arg + single-ticker mode

**File: `src/scraper/main.rs`**

Add `std::env::args()` parsing after env var loading (line ~181). If `--ticker <TICKER>` is present:
1. Connect to DB, run migrations, init config (same as now)
2. Build rate limiters + HTTP client (same as now)
3. Call new `fetch_single_ticker()` instead of `run_all_fetches()`
4. Exit (skip scheduler)

**New function: `fetch_single_ticker()`** in `main.rs` (~50 lines):
```
1. astrology::seed_natal_charts(pool)          // ensure natal chart exists
2. astrology::compute_daily_transits(pool)     // current planet positions
3. astrology::compute_astro_scores(pool)       // compute for all (fast, local math)
4. prices::fetch_and_store(ticker, pool, client, api_key)  // AV price data
5. finnhub: news + recommendations for ticker  // if key present
6. sentiment: AV sentiment for ticker          // if budget allows
7. fundamentals: FMP fetch for ticker          // if key present
8. lagrange::compute_all_scores(pool)          // recompute composites
```

**Visibility changes needed:**
- `src/scraper/prices.rs:60` — `fetch_and_store` is private. Make `pub(crate)`.
- `src/scraper/fundamentals.rs:139` — `fetch_and_store` is private. Make `pub(crate)`.
- `src/scraper/finnhub.rs` — inner `fetch_finnhub_news()` and `fetch_finnhub_recommendations()` are private. Make `pub(crate)`.

### 2c. Dashboard: update handler

**File: `src/dashboard/update/data.rs`**

`FetchThisTicker` handler:
- Set `fetching_ticker = true`, push toast "Fetching data for {ticker}..."
- Locate scraper binary: `std::env::current_exe().parent().join("scraper")` (+ `.exe` on Windows)
- Spawn via `Task::perform` wrapping `tokio::process::Command::new(scraper_path).args(["--ticker", &ticker]).output()`
- On success -> `FetchTickerComplete(Ok(()))`
- On failure -> `FetchTickerComplete(Err(stderr))`

`FetchTickerComplete` handler:
- Set `fetching_ticker = false`
- Ok: push toast "Data fetched!", trigger `RefreshNow` to re-read DB
- Err: set `fetch_ticker_error`, push error toast

## Phase 3: LLM Agent Path (v5.0.3)

**Goal:** Optional LLM-backed analysis via Anthropic Claude API, alongside existing template agents. API key managed in Settings. Graceful fallback to template when LLM unavailable.

**Architecture:** Direct HTTP POST to `api.anthropic.com/v1/messages` via reqwest (already a dependency). Each persona gets a system prompt encoding its philosophy + the AgentContext financial data. Claude returns structured JSON parsed into the existing `AgentAnalysis` struct.

### 3a. New enum + API call function

**File: `src/dashboard/agents.rs`**

New types:
```rust
pub enum AgentMode { Template, Llm }
```

New functions (~200 lines):
- `build_system_prompt(persona, ctx) -> String` — persona philosophy + financial context + JSON response format instructions
- `format_context_for_llm(ctx) -> String` — serialize AgentContext fields to human-readable text
- `pub async fn analyze_llm(persona, ctx, api_key) -> Result<AgentAnalysis, String>` — POST to Anthropic API, parse response
- `parse_llm_response(persona, json) -> Result<AgentAnalysis, String>` — extract JSON from Claude's response (handles markdown-wrapped JSON)

System prompt structure per persona:
- Persona identity + investment philosophy (3-5 sentences)
- Financial context block (all available metrics from AgentContext)
- JSON response format specification (headline, analysis, verdict, key_metrics, astro_take)
- "Stay in character" instruction

Model: `claude-sonnet-4-20250514` (fast, cheap, good for structured output)

### 3b. State changes
